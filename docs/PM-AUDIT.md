## Часть 2. Консультация с экспертом-управляющим СТО

> **Эксперт:** Управляющий СТО, 12 лет опыта, 3 СТО на 15 постов суммарно, штат 40 человек.

### Что эксперт оценил положительно:
1. **Календарь записей с постами** — «Это правильный подход, у нас так же организовано. Визуальное распределение по подъёмникам критично.»
2. **Связка запись → заказ-наряд** — «Часто в программах эти сущности живут отдельно. Связка экономит время.»
3. **Проверка остатков при завершении** — «Молодцы что не дают завершить без запчастей на складе. Это основная боль — списывать нечего.»
4. **Приходные накладные** — «Черновик → проведение — это правильный флоу. Но нужна возможность отмены проведения (сторно).»

### Что эксперт назвал критическими проблемами:

#### ПРОБЛЕМА 1: Нет истории обслуживания автомобиля
> «Клиент приезжает и спрашивает — что мне делали в прошлый раз? Когда менял масло? А у вас нет карточки авто с историей. Это MUST HAVE для любого СТО.»

#### ПРОБЛЕМА 2: Нет учёта рабочего времени
> «Я не знаю, сколько часов мастер работал, сколько простаивал, какой у него KPI. Без табеля я не могу считать зарплату нормально.»

#### ПРОБЛЕМА 3: Отсутствие отчётности
> «У вас нет НИ ОДНОГО отчёта. Выручка за день/неделю/месяц? Средний чек? Загрузка постов? Маржинальность? Я вслепую управляю бизнесом.»

#### ПРОБЛЕМА 4: Нет учёта кассы
> «У вас есть оплата, но нет кассы. Я не знаю, сколько денег в кассе, сколько наличкой пришло, сколько по карте. Нет Z-отчёта, нет инкассации.»

#### ПРОБЛЕМА 5: Склад не связан с заказ-нарядами при создании
> «Когда мастер-приёмщик добавляет запчасть в заказ, он должен ВИДЕТЬ остатки на складе. Сейчас он вводит запчасти вслепую и узнаёт что их нет только при завершении.»

#### ПРОБЛЕМА 6: Нет процесса закупки
> «Когда запчасть заканчивается или нужна для заказа — нет заявки на закупку, нет процесса заказ→доставка→приёмка. Это огромная дыра.»

#### ПРОБЛЕМА 7: Нет печатных форм
> «Мне нужно печатать: заказ-наряд для клиента, акт выполненных работ, накладные, счета. Без бумаг бизнес не работает.»

#### ПРОБЛЕМА 8: Нет SMS/уведомлений клиенту
> «Клиент записался — ему должно прийти SMS с подтверждением. Машина готова — SMS. Напоминание за день до визита. Без этого теряем 15-20% записей.»

#### ПРОБЛЕМА 9: Зарплата — примитивный расчёт
> «5% от базовой за каждый заказ? У нас механики получают % от выработки (от суммы работ), у мастеров-приёмщиков — % от выручки. Нужна гибкая схема мотивации.»

#### ПРОБЛЕМА 10: Нет разделения на цеха/отделы
> «У крупного СТО есть слесарный цех, кузовной, малярный, электрика. Каждый — свои мастера, свои посты, своя загрузка.»

## Часть 3. Workflow — Что нужно изменить в текущем проекте

### 3.1 Критические изменения (блокируют ежедневную работу)

#### W-01: Связать склад с заказ-нарядом в реальном времени
**Текущее:** Запчасти добавляются в заказ без проверки остатков. Проверка только при завершении.
**Нужно:** При добавлении запчасти в заказ — показывать остаток на складе. Предупреждение если остаток < нужного кол-ва. Возможность зарезервировать запчасть.

#### W-02: История обслуживания автомобиля
**Текущее:** Нет.
**Нужно:** Карточка авто → вкладка «История» → все заказ-наряды по этому авто, отсортированные по дате. С возможностью быстро посмотреть что делали, какие запчасти ставили.

#### W-03: Отчёты — минимальный набор
**Текущее:** Нет ни одного отчёта.
**Нужно (MVP):**
- Выручка за период (по дням/неделям/месяцам)
- Средний чек
- Кол-во заказов по статусам
- Загрузка постов (% заполненности слотов)
- Топ-10 работ по выручке
- Топ-10 запчастей по продажам

#### W-04: Dashboard — живые данные
**Текущее:** Захардкоженные значения.
**Нужно:** Реальные KPI из БД: заказов сегодня, выручка сегодня, записей на завтра, позиций с низким остатком.

#### W-05: Страница зарплаты (фронтенд)
**Текущее:** Пустая страница-заглушка, бэкенд есть.
**Нужно:** UI для расчёта зарплаты: выбор сотрудника, период, расчёт, утверждение, выплата. Таблица с историей начислений.

### 3.2 Важные улучшения (влияют на эффективность)

#### W-06: Печатные формы
- Заказ-наряд (для клиента)
- Акт выполненных работ
- Приходная накладная
- Счёт на оплату

#### W-07: Процесс закупки
- Заявка на закупку (из low-stock или из заказа)
- Связка заявка → поставщик → накладная
- Статусы заявки: создана → отправлена → в пути → получена

#### W-08: Касса / учёт денежных потоков
- Текущий остаток кассы
- Приход (оплаты от клиентов)
- Расход (оплата поставщикам, зарплата, прочие расходы)
- Z-отчёт (итоги дня)

#### W-09: Отмена проведения накладной (сторно)
**Текущее:** Проведённую накладную нельзя отменить.
**Нужно:** Возможность сторнировать проведение (откатить остатки, создать обратные транзакции).

#### W-10: Гибкая схема зарплаты
**Текущее:** Фикс + 5% от базовой за заказ.
**Нужно:**
- Механик: оклад + % от суммы работ (нормо-часов)
- Приёмщик: оклад + % от выручки принятых заказов
- Настраиваемые коэффициенты по сотрудникам

### 3.3 Желательные улучшения (конкурентные преимущества)

#### W-11: SMS/Push уведомления
- Подтверждение записи
- Напоминание за день
- «Ваш автомобиль готов»
- Рекламные рассылки

#### W-12: Мобильная версия для механика
- Просмотр своих заказов
- Отметка «начал работу» / «закончил»
- Фото-отчёт по работе

#### W-13: Клиентский портал
- Личный кабинет клиента
- История обслуживания
- Онлайн-запись
- Статус заказа в реальном времени

## Часть 5. Бизнес-требования для Tech Lead

### БТ-01: Живой Dashboard

**Цель:** Руководитель при входе в систему видит текущее состояние бизнеса.

**Требования:**
- API `GET /api/v1/dashboard/stats` — агрегированные данные:
  - `orders_today` — кол-во заказов, созданных сегодня
  - `revenue_today` — выручка за сегодня (сумма succeeded платежей)
  - `revenue_month` — выручка за текущий месяц
  - `appointments_tomorrow` — записей на завтра
  - `low_stock_count` — кол-во позиций ниже min_quantity
  - `orders_in_progress` — заказов в работе
  - `avg_check_month` — средний чек за месяц
- Фронтенд: заменить захардкоженные StatCard на реальные данные
- Обновление данных: при каждом переходе на Dashboard

**Модель данных:** Новый эндпоинт, агрегация из существующих таблиц.

---

### БТ-02: История обслуживания автомобиля

**Цель:** Мастер-приёмщик при приёмке авто видит полную историю обслуживания.

**Требования:**
- API `GET /api/v1/vehicles/{vehicle_id}/history` — список заказов по авто:
  - Номер заказа, дата, статус, механик, список работ, список запчастей, сумма, пробег на момент визита
- Фронтенд: в диалоге заказа, в блоке «Автомобиль» — кнопка «История». Открывает список предыдущих визитов.
- Сохранять пробег в заказе (snapshot на момент создания)

**Модель данных:** Добавить поле `mileage_at_service` в модель `Order`. Новый API endpoint.

---

### БТ-03: Остатки склада в заказ-наряде

**Цель:** При добавлении запчасти в заказ видеть текущий остаток на складе.

**Требования:**
- API `GET /api/v1/parts/?search=...` — расширить ответ полем `stock_quantity` (остаток из warehouse_items)
- Фронтенд: при выборе запчасти в Autocomplete — показывать остаток. Если остаток < quantity, показать предупреждение (жёлтый бадж «Мало» или красный «Нет на складе»)
- Не блокировать добавление — только предупреждать

**Модель данных:** JOIN parts ↔ warehouse_items в API запчастей.

---

### БТ-04: Модуль отчётов

**Цель:** Руководитель может анализировать бизнес-показатели за произвольный период.

**Требования:**
- Новая страница `/reports` с табами:
  1. **Выручка** — график выручки по дням/неделям/месяцам. Фильтр по периоду.
  2. **Заказы** — кол-во заказов по статусам, средний чек, средняя длительность.
  3. **Загрузка постов** — % заполненности слотов по дням. Тепловая карта.
  4. **Топ работ** — топ-10 работ по выручке и кол-ву.
  5. **Топ запчастей** — топ-10 запчастей по продажам и выручке.
  6. **Сотрудники** — выработка по механикам (кол-во заказов, сумма работ).

**Backend:**
- `GET /api/v1/reports/revenue?date_from=&date_to=&group_by=day|week|month`
- `GET /api/v1/reports/orders?date_from=&date_to=`
- `GET /api/v1/reports/post-load?date_from=&date_to=`
- `GET /api/v1/reports/top-works?date_from=&date_to=&limit=10`
- `GET /api/v1/reports/top-parts?date_from=&date_to=&limit=10`
- `GET /api/v1/reports/employees?date_from=&date_to=`

**Фронтенд:** Использовать библиотеку `recharts` или `chart.js` для графиков.

### БТ-05: Касса и учёт денежных потоков

**Цель:** Ежедневный контроль движения денег.

**Требования:**
- Новая модель `CashTransaction`:
  - `id, type (income/expense), category (client_payment/supplier_payment/salary/rent/other), amount, payment_method (cash/card/bank), description, order_id?, employee_id, created_at`
- При создании платежа за заказ — автоматически создавать запись в кассу
- При выплате зарплаты — автоматически создавать расход
- Ручное внесение расходов (аренда, коммунальные, прочее)
- Баланс кассы (наличные / безнал отдельно)
- Z-отчёт: итоги дня по категориям

**Backend:**
- `GET /api/v1/cash/balance` — текущий баланс
- `GET /api/v1/cash/transactions?date_from=&date_to=&type=`
- `POST /api/v1/cash/transactions` — ручной ввод расхода/дохода
- `GET /api/v1/cash/z-report?date=` — Z-отчёт за день

### БТ-06: Заявки на закупку (Purchase Orders)

**Цель:** Систематизировать процесс закупки запчастей.

**Требования:**
- Новая модель `PurchaseOrder`:
  - `id, number, supplier_id, status (draft/sent/in_transit/received/cancelled), created_by, created_at, expected_date, notes`
- Модель `PurchaseOrderLine`:
  - `id, purchase_order_id, part_id, quantity, estimated_price`
- Создание заявки: из low-stock списка (кнопка «Заказать»), или из заказ-наряда (кнопка «Нет на складе — заказать»), или вручную
- При поступлении: создание приходной накладной из заявки (предзаполнение)
- Статусный флоу с уведомлениями

**Backend:**
- CRUD `GET/POST/PUT /api/v1/purchase-orders`
- `POST /api/v1/purchase-orders/{id}/send` — отправить поставщику
- `POST /api/v1/purchase-orders/{id}/receive` — создать накладную

### БТ-08: Гибкая зарплатная схема

**Цель:** Настраиваемая мотивация для разных категорий сотрудников.

**Требования:**
- Новая модель `SalaryScheme`:
  - `id, employee_id, base_salary, works_percentage (% от суммы работ), revenue_percentage (% от выручки), per_order_bonus, effective_from`
- Расчёт зарплаты:
  - Механик: `base + works_sum * works_percentage / 100`
  - Приёмщик: `base + revenue * revenue_percentage / 100`
  - Общее: `+ per_order_bonus * orders_count - penalties`
- Доработать `salary_service.py` для работы с SalaryScheme
- Фронтенд: настройка схемы в карточке сотрудника + полноценная страница зарплаты
