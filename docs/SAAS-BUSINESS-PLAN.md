# AutoService SaaS — Бизнес-план и архитектура платформы

> Дата: 2026-03-12
> Статус: MVP Planning
> Продукт: Облачная платформа управления автосервисом (SaaS)

---

## 1. Резюме продукта

**Что продаём:** Готовое облачное решение для управления автосервисом — записи, наряд-заказы, склад, зарплаты, аналитика.

**Целевая аудитория:** Владельцы автосервисов, СТО, шиномонтажей, детейлинг-студий (РФ/СНГ).

**Модель монетизации:** Подписка (SaaS) с оплатой через ЮKassa.

**Ключевое преимущество:** Быстрый старт — зарегистрировался, получил готовый автосервис-менеджер за 5 минут, без установки и серверов.

### 2.2 Проценка запчастей — интеграция через агрегатор

> **Ответ на вопрос: да, единый API-агрегатор существует. Не нужно делать 150 интеграций.**

#### Варианты API-агрегаторов (один API = десятки/сотни поставщиков)

| Агрегатор | Поставщиков | Цена | API | Подходит для SaaS? |
|-----------|-------------|------|-----|-------------------|
| **QWEP** | сотни | от 25 000 ₽/год (~2 083 ₽/мес) за юзера | ✅ REST API, WEB, 1С | ✅ Да — есть API для сторонних приложений |
| **Sort1** | 140+ | 4 950 ₽/мес (3 юзера) | ✅ API (независимый проценщик) | ✅ Да — единственный независимый проценщик |

#### Расчёт нагрузки на API проценки (средний автосервис, 4-5 подъёмников)

```
Средний автосервис, 4-5 подъёмников:

Заказов в день:           6-10 (1-2 авто на подъёмник/день)
Заказов в месяц:          ~150-200
Запросов на 1 заказ:      ~3-5 (поиск + аналоги + уточнения)
"Просто посмотреть цену": ~200-300 запросов/мес (без заказа)

ИТОГО на 1 тенант:        ~1 000 - 1 500 запросов/мес
```

Это немного — для API это лёгкая нагрузка.

#### Как покупать API — один пакет или по тенантам?

**Покупать подписку QWEP на каждого тенанта отдельно — НЕ ВАРИАНТ:**
```
10 тенантов × 2 083 ₽/мес = 20 830 ₽/мес → убивает маржу
50 тенантов × 2 083 ₽/мес = 104 150 ₽/мес → катастрофа
```

**Правильный подход — партнёрский/OEM договор с оплатой за объём запросов:**

```
┌─────────────┐     ┌──────────────────────────┐     ┌──────────┐
│ Тенант A    │────►│                          │     │          │
│ Тенант B    │────►│  Platform Proxy API      │────►│  QWEP    │
│ Тенант C    │────►│  (наш промежуточный слой)│     │  API     │
│ ...         │────►│                          │     │          │
└─────────────┘     │  • 1 API-ключ на всех    │     └──────────┘
                    │  • Кеширование ответов   │
                    │  • Лимиты на тенант      │
                    │  • Логирование/биллинг   │
                    └──────────────────────────┘
```

**Один API-ключ на всю платформу**, все тенанты ходят через наш прокси-слой.

**Преимущества прокси-слоя:**

| Функция | Зачем |
|---------|-------|
| **Лимиты по тарифу** | Старт = 0 запросов, Бизнес = до 2 000/мес, Про = до 5 000/мес |
| **Логирование** | Знаем сколько каждый тенант потребляет, видим аномалии |
| **Отказоустойчивость** | Если QWEP упал — показываем "проценка временно недоступна", приложение не ломается |
| **Смена провайдера** | Завтра Sort1 станет выгоднее — меняем в одном месте, тенанты не замечают |

> **Важно: кеширование цен НЕ применяется.**
> Цены у поставщиков меняются в реальном времени, у разных автосервисов
> могут быть разные договорные условия. Каждый запрос идёт напрямую в QWEP.
> Все расчёты маржи строятся на полном объёме запросов без экономии.

**Расчёт объёма запросов:**

```
30 тенантов × 1 200 запросов = 36 000 запросов/мес
50 тенантов × 1 200 запросов = 60 000 запросов/мес
100 тенантов × 1 200 запросов = 120 000 запросов/мес
```

#### Что обсудить с QWEP (action item)

Позвонить в QWEP (8-800-500-54-80) и задать конкретные вопросы:

```
"Мы разрабатываем SaaS-платформу для автосервисов.
Хотим встроить вашу проценку через API.
Планируемый объём: 10-100 автосервисов = 10 000 - 150 000 запросов/мес.

1. Есть ли партнёрский/OEM тариф с оплатой за объём запросов
   (а не за количество юзеров)?
2. Можно ли работать через 1 API-ключ на всю платформу?
3. Какая цена за 10К / 50К / 200К запросов в месяц?
4. Есть ли white-label (без отображения бренда QWEP)?
5. Есть ли партнёрская комиссия если наши клиенты
   начнут заказывать запчасти через вас?"
```

Такие компании **заинтересованы** в SaaS-партнёрах: вы приводите десятки клиентов, они дают оптовую цену. Win-win.

**Альтернатива:** Если QWEP не предложит партнёрский тариф — обратиться в Sort1 (4 950 ₽/мес, 140+ поставщиков, позиционируются как независимый проценщик, более гибкие условия).

#### Рекомендация для MVP

1. **MVP (запуск без проценки):** тариф "Старт" 990 ₽ работает сразу, маржа 60%
2. **v1.1 (через 1-2 мес):** интеграция QWEP после согласования партнёрского тарифа
3. Срок интеграции: 2-3 недели (прокси-слой + UI поиска в заказ-наряде)

**Бизнес-модель проценки в тарифе:**
- Тариф "Старт" (990 ₽) — проценка НЕ входит (мотивация перейти на "Бизнес")
- Тариф "Бизнес" (2 990 ₽) и "Про" (4 990 ₽) — проценка включена
- Кеширование цен НЕ применяется (цены меняются в реальном времени, разные условия у разных клиентов)
- Себестоимость: 1 000-1 500 ₽/мес за тенант (зависит от тарифа QWEP)
- Маржа: 30-47% (Бизнес), 46-58% (Про) — зависит от результата переговоров с QWEP

### 2.3 Пробный период (Trial)

- **10 дней** бесплатно, тариф «Бизнес» (включая проценку)
- Не требует ввода карты при регистрации (снижает барьер входа)
- За 3 дня и за 1 день до окончания — email/SMS-напоминание
- После окончания — аккаунт блокируется (данные сохраняются 30 дней)
- Клиент может оплатить в любой момент и разблокировать доступ

### 2.4 Жизненный цикл клиента

```
Лендинг → Регистрация → Trial (10 дней) → Оплата → Активная подписка
                                              ↓                ↓
                                     Блокировка ←── Не продлил (grace 3 дня)
                                              ↓
                                     Данные хранятся 30 дней → Удаление
```

### 2.5 Дополнительные источники дохода

1. **Онбординг за деньги** — помощь с настройкой, импорт данных из Excel (разовая услуга 5 000–15 000 ₽)
2. **SMS-пакеты** — перепродажа SMS для напоминаний клиентам (маржа 30-50%)
3. **Интеграции как допуслуга** — подключение к 1С, маркетплейсам запчастей
4. **White-label** — для франшиз автосервисов (кастомный домен, логотип)
5. **Реферальная программа** — 1 месяц бесплатно за привлечённого клиента
6. **Комиссия с проценки** (v2) — при переходе на ЮKassa Marketplace можно брать % с заказов запчастей

### 3.2 Выбор стратегии мультитенантности

**Ваша идея:** Каждый клиент = отдельный Docker-контейнер + отдельная БД.

**Анализ подходов:**

| Подход | Плюсы | Минусы | Когда |
|--------|-------|--------|-------|
| **Отдельные контейнеры** (ваш вариант) | Полная изоляция, простой код, легко удалять | Дорого по RAM (~200-300 МБ на тенант), медленный провижининг | до 50-100 клиентов |
| **Общее приложение, отдельные БД** | Экономия RAM, быстрее | Нужен connection routing | 100-500 клиентов |
| **Общая БД, tenant_id** | Максимальная экономия | Сложный код, риск утечки данных | 500+ клиентов |

**Рекомендация для MVP:** Ваш вариант (отдельные контейнеры) — правильный выбор на старте. Он самый простой в реализации и даёт полную изоляцию данных клиентов. На 1 сервере с 32 ГБ RAM уместится ~50-80 клиентов. При росте можно мигрировать на подход с общим приложением.

#### 3.3.2 Platform API (мозг платформы)

Новый FastAPI-сервис, отвечающий за всё, что НЕ является функциональностью автосервиса:

```

│   ├── main.py
│   ├── config.py
│   ├── database.py              # Platform DB (PostgreSQL)
│   ├── models/
│   │   ├── owner.py             # Владелец аккаунта (кто платит)
│   │   ├── tenant.py            # Инстанс автосервиса
│   │   ├── subscription.py      # Подписка (тариф, статус, даты)
│   │   ├── payment.py           # Платежи через ЮKassa
│   │   └── invoice.py           # Счета
│   ├── api/
│   │   ├── auth.py              # Регистрация, вход, восстановление пароля
│   │   ├── billing.py           # Тарифы, оплата, история
│   │   ├── tenants.py           # Управление инстансами
│   │   ├── webhooks.py          # ЮKassa webhooks
│   │   └── admin.py             # Админ-панель API
│   ├── services/
│   │   ├── provisioning.py      # Создание/удаление Docker-контейнеров
│   │   ├── yookassa_service.py  # Работа с ЮKassa API
│   │   ├── notification.py      # Email/SMS уведомления
│   │   └── scheduler.py         # Крон-задачи (проверка подписок)

## 4. Интеграция с ЮKassa

### 4.1 Процесс оплаты

```
Клиент нажимает "Оплатить" в личном кабинете
        │
        ▼
Platform API → POST https://api.yookassa.ru/v3/payments
        │      {
        │        "amount": {"value": "3990.00", "currency": "RUB"},
        │        "confirmation": {"type": "redirect", "return_url": "..."},
        │        "capture": true,
        │        "description": "Подписка AutoService Бизнес, март 2026",
        │        "metadata": {"subscription_id": "...", "owner_id": "..."}
        │      }
        │
        ▼
Редирект клиента на ЮKassa → Клиент вводит данные карты
        │
        ▼
ЮKassa → POST /api/webhooks/yookassa (webhook)
        │      {"event": "payment.succeeded", ...}
        │
        ▼
Platform API:
  1. Верифицирует подпись webhook
  2. Обновляет payment.status → succeeded
  3. Продлевает подписку (current_period_end += 30 дней)
  4. Если тенант suspended → активирует его (запуск контейнера)
  5. Отправляет email-чек клиенту
```

### 4.2 Автоплатежи (рекурентные)

ЮKassa поддерживает автоплатежи через сохранённые методы:

1. При первой оплате — `save_payment_method: true`
2. ЮKassa возвращает `payment_method_id`
3. При следующем списании — создаём платёж с `payment_method_id` без участия клиента
4. Если списание не прошло — 3 попытки с интервалом 1/3/7 дней (grace period)

### 4.3 Чеки и 54-ФЗ

ЮKassa умеет отправлять чеки автоматически (онлайн-касса), для этого нужно:
- Передавать `receipt` объект при создании платежа
- Указать ИНН вашей компании, систему налогообложения
- Это **обязательно** по закону РФ для онлайн-продаж

## 5. Приём платежей внутри тенанта (автосервис → автовладелец)

> Это ОТДЕЛЬНЫЙ контур оплаты от платформенного биллинга.
> Платформа принимает оплату за подписку от владельца автосервиса.
> Тенант принимает оплату за ремонт от клиентов автосервиса (автовладельцев).

### 5.1 Два уровня платежей в системе

```
┌──────────────────────────────────────────────────────────┐
│  УРОВЕНЬ 1: Платформа (вы → владелец автосервиса)        │
│  ЮKassa аккаунт: ВАШ                                     │
│  Назначение: подписка на SaaS                             │
│  Пример: "ООО АвтоПлюс оплатил подписку 3 990 ₽/мес"   │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│  УРОВЕНЬ 2: Тенант (автосервис → автовладелец)           │
│  ЮKassa аккаунт: КЛИЕНТА (владельца автосервиса)         │
│  Назначение: оплата за ремонт, запчасти, услуги          │
│  Пример: "Иванов И.И. оплатил наряд-заказ №142: 15 400₽"│
└──────────────────────────────────────────────────────────┘
```

### 5.2 Варианты архитектуры приёма платежей в тенанте

| Подход | Описание | Плюсы | Минусы |
|--------|----------|-------|--------|
| **A. У каждого клиента свой ЮKassa** | Владелец автосервиса сам подключает ЮKassa, вводит shop_id и secret_key в настройках | Просто для вас, деньги идут напрямую клиенту, нет юридических рисков | Клиенту нужно самому подключать ЮKassa (барьер) |
| **B. Вы — платёжный агрегатор** | Все платежи идут через ваш ЮKassa, вы распределяете деньги клиентам | Удобно для клиента | Нужна лицензия НКО/банковская, огромные юридические риски, 115-ФЗ |
| **C. ЮKassa Marketplace (сплитование)** | ЮKassa Split — платёж делится между вами и продавцом | Легальный способ брать комиссию | Нужен договор с ЮKassa как маркетплейс, сложная интеграция |

**Рекомендация для MVP: Подход A** — каждый клиент подключает свой ЮKassa-аккаунт.

Причины:
- Нулевые юридические риски для вас (вы не касаетесь чужих денег)
- Самая простая реализация
- Клиенту не сложно подключить ЮKassa (ИП/ООО у владельца автосервиса уже есть)
- В будущем можно добавить подход C (маркетплейс) для доп. монетизации

### 5.3 Как это работает — важное уточнение

> **Мы НЕ подключаем ЮKassa за клиента автоматически.**
> ЮKassa — это договор между ЮKassa и юрлицом клиента. Мы в этом процессе не участвуем.
> Наше приложение — это интерфейс, который использует ЮKassa API **от имени клиента** с его ключами.
> Аналогия: Tilda, Shopify, Битрикс — они не подключают эквайринг за тебя. Ты сам регистрируешься в платёжке, получаешь ключи, вставляешь их в настройках.

#### Процесс подключения глазами клиента (владельца автосервиса)

```
Шаг 1 (вне нашей системы): Клиент САМОСТОЯТЕЛЬНО
  → Идёт на yookassa.ru
  → Регистрируется как магазин (нужно ИП или ООО)
  → Загружает документы (ИНН, ОГРН/ОГРНИП, паспорт)
  → Проходит верификацию ЮKassa — занимает 1-5 рабочих дней
  → Получает shop_id и secret_key в личном кабинете ЮKassa
  → Настраивает webhook URL: https://{slug}.autoservice.app/api/v1/webhooks/yookassa

Шаг 2 (в нашей системе): Клиент заходит в приложение
  → Настройки → Приём онлайн-платежей
  → Вводит shop_id и secret_key
  → Нажимает "Проверить подключение" (тестовый запрос к ЮKassa API)
  → Если ОК — статус: "✅ Подключено"
  → Заполняет данные для чеков (ИНН, система налогообложения)
  → Готово — кнопка "Онлайн-оплата" появляется в заказах
```

**Что мы делаем для клиента, чтобы упростить процесс:**
- Пошаговая инструкция в приложении: "Как подключить ЮKassa за 5 минут"
- Скриншоты: где взять shop_id и secret_key в ЛК ЮKassa
- Автоподстановка webhook URL (клиенту остаётся только скопировать и вставить в ЛК ЮKassa)
- Кнопка "Проверить подключение" — мгновенная обратная связь, всё ли настроено правильно

#### Будущее: Упрощённое подключение через ЮKassa Marketplace API (v2)

В перспективе (когда будет 50+ клиентов) можно заключить с ЮKassa договор как маркетплейс. Тогда:

```
Клиент в нашем приложении:
  → Нажимает "Подключить приём платежей"
  → Заполняет данные ИП/ООО прямо в нашей форме
  → Мы через ЮKassa Marketplace API отправляем заявку на создание "суб-магазина"
  → ЮKassa верифицирует и создаёт суб-магазин (1-3 дня)
  → Платежи проходят через наш аккаунт с автоматическим сплитом
  → Мы можем брать комиссию (например, 1-2% с каждого платежа)

Требования:
  - Договор с ЮKassa как маркетплейс-партнёр
  - Более сложная интеграция (Marketplace API)
  - Имеет смысл при масштабе (оправдывает затраты на интеграцию)
```

### 5.5 Онлайн-касса (54-ФЗ) — что нужно знать

**Зачем:** Любой бизнес, принимающий оплату от физлиц, обязан выдавать кассовый чек.

**Три сценария фискализации для тенанта:**

| Сценарий | Описание | Для кого |
|----------|----------|----------|
| **ЮKassa + встроенная касса** | При онлайн-оплате ЮKassa сама отправляет чек (нужно передать `receipt` в API) | Онлайн-оплаты |
| **Облачная касса (АТОЛ Онлайн, OrangeData)** | Для чеков по наличным и карте (терминал). API-интеграция | Оффлайн-оплаты |
| **Физическая касса (Эвотор, АТОЛ)** | Автосервис использует свою ККТ, в нашей системе только учёт | Если уже есть касса |

**Рекомендация для MVP:**
1. Онлайн-оплаты → чек через ЮKassa (передаём `receipt` объект) — **реализуем**
2. Наличные/карта → пометка в системе "оплачено наличными/картой", чек пробивает сам клиент на своей кассе — **не усложняем**
3. В будущем → интеграция с облачной кассой (АТОЛ Онлайн) для полной автоматизации

## 6. Управление подписками и блокировки

### 6.1 Состояния подписки

```
                    ┌────────────────────────────┐
                    │                            │
                    ▼                            │
    ┌─────────┐  оплата  ┌────────┐  продление  │
    │  TRIAL  │────────►  │ ACTIVE │─────────────┘
    └────┬────┘           └───┬────┘
         │                    │
    истёк триал          не оплатил
         │                    │
         ▼                    ▼
    ┌──────────┐       ┌───────────┐
    │ EXPIRED  │       │ PAST_DUE  │ (grace 3 дня)
    └────┬─────┘       └─────┬─────┘
         │                   │
         │              не оплатил
         ▼                   ▼
    ┌───────────────────────────┐
    │       SUSPENDED           │  ← контейнер остановлен
    └────────────┬──────────────┘
                 │
            30 дней без оплаты
                 │
                 ▼
    ┌───────────────────────────┐
    │       TERMINATED          │  ← данные удалены
    └───────────────────────────┘
```

### 6.2 Проверка подписки (middleware)

При каждом запросе к тенанту:
1. Nginx делает auth_request к Platform API
2. Platform API проверяет: `subscription.status IN ('trial', 'active')`
3. Результат кешируется в Redis на 5 минут
4. Если подписка неактивна → HTTP 402 → фронтенд показывает "Подписка истекла, оплатите"

### 6.3 Cron-задачи (Scheduler)

| Задача | Расписание | Действие |
|--------|-----------|----------|
| Проверка окончания триала | каждый час | Если trial_ends_at < now → status = expired, suspend tenant |
| Проверка продления подписки | каждый час | Если current_period_end < now → status = past_due |
| Grace period | ежедневно | Если past_due > 3 дней → suspend tenant |
| Напоминание (за 3 дня) | ежедневно | Email: "Подписка заканчивается через 3 дня" |
| Напоминание (за 1 день) | ежедневно | Email: "Подписка заканчивается завтра" |
| Удаление данных | еженедельно | Если suspended > 30 дней → terminate, удалить данные |
| Автосписание | ежедневно | Попытка рекурентного платежа для past_due |

## 7. Безопасность и изоляция

### 7.1 Изоляция тенантов

- **Сеть:** Каждый контейнер в отдельной Docker network, доступ только через reverse proxy
- **БД:** Отдельная PostgreSQL database на каждого тенанта, отдельный DB user
- **Файловая система:** Volumes изолированы, тенанты не имеют доступа к хосту
- **Секреты:** Отдельный SECRET_KEY для JWT каждого тенанта (генерируется при провижининге)

### 7.2 Платформенная безопасность

- HTTPS (Let's Encrypt wildcard сертификат для *.autoservice.app)
- Rate limiting на Platform API
- Webhook signature verification (ЮKassa)
- Логирование всех критичных операций (platform_events)
- Бекапы БД каждые 6 часов

### 7.3 Что вы упускаете — юридические вопросы

1. **Оферта** — публичный договор на сайте (обязательно)
2. **Политика конфиденциальности** — обработка персональных данных (152-ФЗ)
3. **Согласие на обработку ПД** — чекбокс при регистрации
4. **Онлайн-касса** — 54-ФЗ, чеки при каждой оплате (ЮKassa решает)
5. **ИП или ООО** — нужно юрлицо для приёма платежей через ЮKassa
6. **Оператор ПД** — вы храните данные клиентов ваших клиентов (нужно уведомление в Роскомнадзор)

**Итого инфраструктура:** ~12 000–18 000 ₽/мес

```
0-50 клиентов:   1 сервер, отдельные контейнеры
50-200 клиентов:  2-3 сервера, Docker Swarm
200-500 клиентов: Переход на shared app + separate DB
500+ клиентов:    Kubernetes + shared DB с tenant_id

```
Git push → GitHub Actions:
  1. Тесты (pytest + vitest)
  2. Build Docker images
  3. Push to Container Registry
  4. Rolling update всех tenant-контейнеров
     (обновляем по 10 штук за раз, с health-check)
```

**Важный момент:** При обновлении приложения нужно обновить ВСЕ контейнеры тенантов + запустить миграции в КАЖДОЙ БД. Это нетривиальная задача — нужен скрипт массового обновления.

## 9A. Стратегия дифференциации — чем выделиться

### Проблема конкурентов (подтверждена исследованием)

> АвтоДилер и STOCRM — **перекрученные**. Много фич, но сложный интерфейс.
> "Системы проектируют 30-летние разработчики, а используют их 45-летние мастера" (Habr).
> Ваш личный опыт с АвтоДилером это подтверждает.

### Ваше конкурентное преимущество №1: ОНЛАЙН-ОПЛАТА В БАЗОВОМ ТАРИФЕ

> Это **самое важное** конкурентное преимущество для малого автосервиса.

```
АвтоДилер:  Терминал/СБП в точке (клиент платит физически в сервисе)
            ≠ отправка ссылки клиенту на телефон
Splus:      Нет онлайн-оплаты вообще
STOCRM:     Неясно, встроено ли в базу

Вы:         Нажал кнопку → клиент получил ссылку на телефон →
            заплатил из дома / до приезда / после выезда →
            чек 54-ФЗ сформировался автоматически.
```

**Почему это важнее проценки запчастей:**
- Мелкий автосервис (2-3 подъёмника) работает с 2-3 постоянными поставщиками.
  Они и так знают их цены и делают проценку прямо на сайте поставщика.
- А вот "как взять деньги дистанционно / не держать много наличных / не ставить
  кассовый аппарат за 30 000 ₽" — это реальная боль КАЖДОГО малого сервиса.

**Аргумент для продаж:**
```
"Вы отдаёте машину клиенту, он говорит 'переведу завтра' — и пропадает.
 С нами: нажали одну кнопку — клиент получил ссылку — заплатил прямо сейчас.
 Всё автоматически, без терминала, без объяснений бухгалтеру."
```

### Ваше конкурентное преимущество №2: ПРОСТОТА

```
Конкуренты:  100 кнопок на экране, обучение 2 недели, нужен интегратор
Вы:          Открыл → понятно → работаешь. Без инструкций.
```

Это не слабость ("у нас меньше фич"), это **позиционирование**:
- Apple победил не количеством фич, а простотой
- Notion победил Confluence простотой
- Telegram победил сложные мессенджеры простотой

### Ваше конкурентное преимущество №3: КАНБАН-ДОСКА ЗАПИСЕЙ

Ни один конкурент (STOCRM, Splus, АвтоДилер) не имеет визуальной канбан-доски с drag-and-drop для управления записями по постам. Это **уникальная фича**.

У конкурентов:                    У вас:

│ Список записей   │              │ Пост 1 │ │ Пост 2 │ │ Пост 3 │

│ (скучный список) │              │┌──────┐│ │        │ │        │
└──────────────────┘              ││11:00 ││ │        │ │        │
                                  ││Сидоров│ │        │ │        │

**Это нужно сделать главным элементом маркетинга.** Скриншот канбан-доски на лендинг.

### Ваше конкурентное преимущество №4: АНАЛИТИКА С ПЕРВОГО ДНЯ

Текущий дашборд — реально сильный. Сравнение периодов, KPI механиков, прогноз загрузки, алерты. У многих конкурентов аналитика — слабое место или доступна только на дорогих тарифах.

### Ваше конкурентное преимущество №5: БЫСТРЫЙ СТАРТ

```
Конкуренты:                       Вы:
Заявка → Менеджер перезвонит →    Регистрация → 30 секунд →
Демо → Договор → Внедрение →     Готово. Работайте.
Обучение → 2 недели → Работа     (без звонков, без менеджеров)

│                                                          │
│   Начните за 5 минут. Бесплатно 10 дней.                │
│                                                          │
│   [Попробовать бесплатно]                                │
│                                                          │
│   "Клиент сказал 'переведу завтра'?                      │

## 12. Анализ рынка и конкурентов

### 12.1 Рынок автосервисов в России — цифры

| Показатель | Значение | Источник |
|-----------|----------|----------|
| Количество автосервисов в РФ | **~130 200** точек (конец 2025) | АВТОСТАТ |
| Объём рынка автосервисных услуг | **1,2 трлн ₽** (2025) | АВТОСТАТ |
| Доля независимых автосервисов | **32,1%** (~41 800 точек) | АВТОСТАТ |

**Вывод:** Рынок огромный (130К точек), но целевая аудитория для облачного SaaS — это в первую очередь **независимые автосервисы (~42 000)**, которые не привязаны к дилерским стандартам. Даже 0.1% рынка = 42 клиента, что уже окупает инфраструктуру.

### 12.2 Конкуренты — детальный разбор

| Конкурент | Тип | Цена/мес | Сильные стороны | Слабые стороны | Угроза |
|-----------|-----|---------|-----------------|----------------|--------|
| **STOCRM** | Облачная CRM | от ~3 000 ₽ (от 3 мес) | Лидер рынка, полный функционал, мобильное приложение, проценка у 150+ поставщиков, онлайн-касса | Сложный интерфейс, привязка от 3-6 мес, дорого для мелких | **Высокая** |
| **Splus** | Облачная CRM | от 2 990 ₽ (безлимит юзеров) | Хороший функционал, без ограничений по пользователям | Менее известен, UI средний | **Средняя** |
| **1С:Автосервис** | Десктоп/Облако | от 5 000–15 000 ₽ + внедрение | Мощнейший функционал, интеграция с 1С:Бухгалтерия | Очень сложно, дорогое внедрение (50-200К), невозможно внедрить самому | **Низкая** (другой сегмент) |
| **Битрикс24** | Универсальная CRM | от 0 ₽ (базовый) | Бесплатный старт, телефония, ИИ | Не заточена под автосервис, нужна кастомизация | **Низкая** |

| **АвтоДилер Онлайн** | Облачная CRM | по запросу | Каталоги, проценка, WhatsApp/Telegram | Ориентирован на крупных | **Низкая** |

### 12.3 Что реально происходит на рынке — инсайты

**Факт 1: РемОнлайн ушёл — образовалась ниша.**
РемОнлайн был одним из самых популярных и доступных решений. С марта 2022 он заблокирован в РФ и не принимает платежи от российских пользователей. Тысячи автосервисов ищут замену.

**Факт 2: Большинство решений перегружены.**
Реальный кейс на Habr: владелец автосервиса создал свою CRM, потому что Bitrix24, amoCRM и аналоги были слишком сложны для мастеров 40-50 лет. Набрал 135 пользователей органически, без маркетинга. Доход его автосервиса вырос на 40%.

> "Системы проектируют 30-летние разработчики, а используют их 45-летние мастера"

**Факт 3: Рынок НЕ насыщен.**
130 000 автосервисов в РФ. Даже у лидера STOCRM — тысячи, а не десятки тысяч клиентов. Уровень цифровизации малых автосервисов крайне низкий. Многие до сих пор ведут учёт в Excel или блокноте.

**Факт 4: Сокращение рынка на 6% в 2025 — это плюс для SaaS.**
Когда рынок сжимается, выживают те, кто эффективнее. Автоматизация становится не "хотелкой", а необходимостью.

### 12.4 Честная оценка — стоит ли входить?

#### Аргументы ЗА

| # | Аргумент | Вес |
|---|----------|-----|
| 1 | Огромный рынок (130К точек), низкая цифровизация | Сильный |
| 2 | РемОнлайн ушёл из РФ — ниша открыта | Сильный |
| 3 | Существующие решения перегружены и сложны | Сильный |
| 4 | У вас уже есть **готовый продукт** (не с нуля) | Критичный |
| 5 | Вход дешёвый (12-18К ₽/мес на инфраструктуру) | Средний |
| 6 | Рекуррентная выручка (SaaS — предсказуемый доход) | Сильный |
| 7 | Кейс на Habr: 135 юзеров без маркетинга = спрос есть | Средний |

#### Аргументы ПРОТИВ

| # | Аргумент | Вес |
|---|----------|-----|
| 1 | STOCRM и Splus уже закрепились, у них маркетинг и база | Средний |
| 2 | Целевая аудитория (40-50 лет) — сложно продавать онлайн | Средний |
| 3 | Нужна поддержка 24/7 (автосервисы работают рано/допоздна) | Слабый |
| 4 | Один разработчик = медленное развитие фич | Средний |
| 5 | Конверсия trial→paid в B2B SaaS обычно 5-15% | Слабый |

#### Вердикт

**Входить — ИМЕЕТ СМЫСЛ.** Но не из-за отсутствия конкуренции (она есть), а потому что:

1. **У вас уже есть продукт.** Вы не начинаете с нуля. Это экономит 3-6 месяцев разработки.
2. **Ниша после ухода РемОнлайн.** Доступное, простое, облачное решение для небольших автосервисов — это тот сегмент, который недозаполнен.
3. **Ваш конкурент — не STOCRM.** Ваш реальный конкурент — Excel и блокнот. 80% малых автосервисов ещё НЕ автоматизированы.
4. **Низкий порог входа.** 12-18К ₽/мес на инфраструктуру. 5 платящих клиентов = окупаемость. Это не стартап с burn rate в миллионы.

**Ключевое условие успеха:** Не пытайтесь конкурировать с STOCRM по количеству фич. Конкурируйте по **простоте**. "Программа для автосервиса, которую освоит любой мастер за 15 минут".

## 13. Маркетинг и привлечение клиентов (MVP)

### 13.1 Расчёт точки безубыточности

```
Постоянные расходы (месяц 1-2, до роста):
  Сервер (VPS для Platform API + первые тенанты): ~5 000 ₽
  ЮKassa комиссия (3.5%):                           входит в юнит-экономику
  Разработка (ваше время — не считаем):              0 ₽
  ─────────────────────────────────────────────────────────
  Минимальный порог расходов:                   ~5 000 ₽/мес

Прибыль на 1 клиента (средний тариф Бизнес 2990₽):
  Доход:                                         2 990 ₽
  QWEP (стандарт):                              -1 500 ₽
  Сервер (доля):                                  -300 ₽
  ЮKassa + налог:                                 -284 ₽
  ─────────────────────────────────────────────────────────
  Прибыль/клиент:                                ~906 ₽

Нужно клиентов для безубыточности: 5 000 / 906 ≈ 6 клиентов
(если все на Бизнес; при смешанных тарифах — ~8-10 клиентов)

Для выхода на 50 000 ₽/мес чистыми:
  (50 000 + 5 000) / 906 ≈ 61 клиент (оптимистично, без QWEP-партнёрки)
  С QWEP-партнёрским тарифом: ~40 клиентов
```

### 13.2 Почему Яндекс.Директ не окупается на старте

```
Яндекс.Директ — "программа для автосервиса":
  Бюджет/мес:                          30 000 ₽
  Кликов (CPC ~100-200 ₽):             150-300 кликов
  Конверсия лендинг → trial:           3-5%
  Trial-пользователей:                 5-15 чел
  Конверсия trial → paid:              20-30%
  Платящих клиентов в месяц:           1-4 чел
  ─────────────────────────────────────────────────────────
  CAC (стоимость привлечения):         7 500–30 000 ₽
  Средний чек 2 990 ₽/мес × 12 мес = 35 880 ₽ LTV
  Срок окупаемости CAC:                3-10 месяцев

ВЫВОД: Директ работает, но только при накопленной базе (50+ клиентов),
       чтобы конвертировать brand awareness в продажи.
       На старте — cash flow отрицательный.
```

### 13.3 Стратегия месяц 1–2: прямой обзвон (без рекламного бюджета)

> Это единственный способ получить первых 10-20 клиентов за разумные деньги
> и получить обратную связь от реальных пользователей.

```

  → 2ГИС: поиск "автосервис" по вашему городу/региону — скачать список
  → Авито: объявления "автосервис" или "СТО" — прямые контакты
  → Яндекс.Карты: парсинг через Яндекс API (бесплатный лимит)

  Реальный план месяц 2: ещё 500 звонков + онбординг trial → ~5-10 платящих

## 14. Риски и как их митигировать

### 14.1 Технические и бизнес-риски

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Долгий провижининг контейнеров | Средняя | Высокое | Pre-built images, кеширование |
| Утечка данных между тенантами | Низкая | Критическое | Полная изоляция (отдельные контейнеры + БД) |
| Падение сервера = все клиенты offline | Средняя | Критическое | Автоматические бекапы, failover на 2-й сервер |
| Низкая конверсия trial→paid | Высокая | Высокое | Онбординг-wizard, email-цепочка, звонок менеджера |
| Клиенты требуют кастомизацию | Высокая | Среднее | Тариф "Про" с настройками, не делать custom dev |
| Проблемы с ЮKassa (блокировка) | Низкая | Высокое | Подключить 2-й эквайринг (Robokassa, CloudPayments) |
| Обновление 100+ контейнеров | Средняя | Среднее | Blue-green deployment, постепенный rollout |

### 14.2 Юридические риски для владельца платформы

> **Контекст:** Платформа предоставляет финансовый модуль (касса, учёт денежных потоков).
> Клиент ведёт внутренний учёт через наше приложение. Ниже — анализ рисков и способы их закрыть.

#### Ключевой принцип: мы — инструмент, не бухгалтер

Мы продаём **инструмент учёта**, а не оказываем бухгалтерские услуги. Это принципиальное различие:

- **1С, Excel, Google Sheets, Notion** — никто из них не несёт ответственности за то, что клиент туда вписал.
- Мы — в той же категории. Платформа предоставляет интерфейс; что клиент вносит и как использует данные — его ответственность.

#### Когда риски для нас теоретически появляются

| Ситуация | Есть ли риск | Комментарий |
|----------|-------------|-------------|
| Клиент ведёт "серые" схемы через наше приложение | ❌ Нет (при наличии ToS) | Мы не знаем и не контролируем содержимое |
| Налоговая запросила данные конкретного клиента | ⚠️ Процедурный вопрос | Отвечать по закону РФ, нужен юрист |
| Клиент позиционирует нас как "официальную бухгалтерию" | ⚠️ Риск репутации | Чётко указывать в ToS: мы НЕ система бухучёта |
| Мы **знали** о конкретной незаконной схеме и помогали | ✅ Риск есть | Но это уже уголовный контекст, не SaaS |
| Мы — оператор персональных данных | ✅ Риск есть | Регулируется 152-ФЗ, закрывается документами |

#### Что защищает нас: обязательный пакет документов

Перед запуском финансового модуля (и вообще до продакшена) необходимо:

**1. Пользовательское соглашение (Terms of Service)**

Обязательные пункты:
- Сервис является инструментом **внутреннего учёта** и **не является системой бухгалтерского или налогового учёта** в соответствии с законодательством РФ
- Пользователь **самостоятельно несёт ответственность** за достоверность вносимых данных
- Платформа **не несёт ответственности** за использование данных в противоправных целях
- Прямой запрет на использование платформы для незаконной деятельности
- Ограничение ответственности платформы (liability limitation clause)

**2. Политика конфиденциальности (Privacy Policy)**

- Описание того, какие данные собираются и как хранятся
- 152-ФЗ: обязателен раздел об обработке персональных данных
- Уведомление в Роскомнадзор о статусе оператора ПД (бесплатно, через госуслуги)

**3. Договор-оферта**

- Публичный договор на сайте, описывающий условия подписки
- Момент акцепта (нажатие "Зарегистрироваться" = согласие с условиями)

> **Стоимость:** Юрист по IT-праву подготовит полный пакет за ~15 000–30 000 ₽ разово.
> Это обязательная инвестиция — без этих документов продавать SaaS с финансовыми данными рискованно.

#### Про запросы от налоговой и проверяющих органов

Если придёт официальный запрос на данные клиента — это юридически регламентированный процесс:
- Запрос должен быть оформлен официально (на бумаге, с подписью и печатью)
- Наш ответ зависит от типа запроса и нашего юрлица
- **Без юриста на такие запросы не отвечать** — обратиться к юристу перед ответом

**Архитектурное преимущество (изоляция тенантов):** У каждого клиента отдельный контейнер и отдельная БД. Это значит, что даже технически мы не можем "случайно" выдать данные одного клиента другому. Запрос на данные конкретного тенанта затрагивает только его изолированную БД.

#### Про финансовый модуль (касса) — специфические риски

Клиент переходит с Excel и начинает вести учёт в нашей системе. Сценарий «начальный остаток»:
- Пользователь создаёт операцию с категорией **"Начальный остаток"** при первом входе
- Это стандартная практика всех учётных систем (1С, Мой склад, и т.д.)
- Достоверность этой суммы — полностью ответственность пользователя, не наша

**Важно:** Мы не являемся:
- Оператором фискальных данных (ОФД) — это отдельная лицензия
- Системой бухгалтерского учёта по нормам ПБУ
- Налоговым агентом клиента

Мы — **инструмент внутреннего управленческого учёта**. Это нужно явно зафиксировать в ToS и в интерфейсе (например, дисклеймер в разделе "Касса": "Данные носят справочный характер и не являются официальной бухгалтерской отчётностью").

## Инфраструктура: Автоматическое развёртывание тенантов и DNS

### Принцип работы доменов

Для изолированных инстансов каждого клиента используется **wildcard DNS-запись** — настраивается один раз на регистраторе и больше не трогается.

**Запись на регистраторе (один раз навсегда):**
```
Тип: A
Имя: *
Значение: <IP сервера>
```

После этого любой поддомен автоматически резолвится на наш сервер без ручных правок:
```
client1.yourdomain.com  → IP сервера  ✅ автоматически
client2.yourdomain.com  → IP сервера  ✅ автоматически
anyname.yourdomain.com  → IP сервера  ✅ автоматически
```

### Маршрутизация на сервере

Все запросы приходят на один IP. Reverse proxy (Traefik) смотрит на заголовок `Host` и направляет в нужный контейнер:

```
Host: client1.yourdomain.com → Docker-контейнер клиента 1
Host: client2.yourdomain.com → Docker-контейнер клиента 2
```

Traefik подхватывает новые контейнеры автоматически через labels — перезапуск/правка конфигов не нужны.

### Что происходит при регистрации клиента

```
1. Клиент регистрируется на лендинге
2. Platform API:
   - Создаёт запись в БД (tenant_id, subdomain)
   - Поднимает Docker-контейнер для тенанта
   - Traefik автоматически подхватывает новый роут
3. Клиент получает доступ: client1.yourdomain.com
```

### SSL-сертификаты

Используется **Let's Encrypt wildcard certificate** (`*.yourdomain.com`) — один сертификат покрывает все поддомены, автообновление через certbot/acme.

### Итог: что настраивается руками (один раз)

| Действие | Частота |
|----------|---------|
| Wildcard DNS `*.yourdomain.com → IP` | Один раз навсегда |
| Wildcard SSL сертификат | Один раз, далее автообновление |
| Установка Traefik на сервер | Один раз |

Регистратор после первоначальной настройки из процесса выбывает полностью.
