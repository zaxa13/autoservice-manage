# БТ-05: Касса и учёт денежных потоков — Тест-кейсы

> Дата: 2026-03-20  
> Статус: Готово к тестированию  
> Роли для тестирования: admin, accountant

**Базовый URL:** `http://localhost:8000`  
**Swagger UI:** `http://localhost:8000/docs`  
**Frontend:** `http://localhost:5173/cashflow`

---

## Подготовка к тестированию

1. Запустить бэкенд и фронтенд
2. Войти под ролью **admin** (имеет доступ ко всем операциям)
3. Перейти в раздел **Касса** в боковом меню

## Блок 1: Счета (Accounts)

### TC-01 — Создание счёта «Касса (наличные)»

**Предусловие:** авторизован как admin  
**Шаги:**
1. `POST /api/v1/cashflow/accounts`
```json
{
  "name": "Касса (наличные)",
  "account_type": "cash",
  "initial_balance": 35000
}
```

**Ожидаемый результат:**
- HTTP 201
- Поле `current_balance` = 35000 (равно `initial_balance`, операций ещё нет)
- `is_active` = true

---

### TC-02 — Создание банковского счёта

**Шаги:**
1. `POST /api/v1/cashflow/accounts`
```json
{
  "name": "Расчётный счёт Сбербанк",
  "account_type": "bank",
  "initial_balance": 0
}
```

**Ожидаемый результат:**
- HTTP 201
- `account_type` = "bank"

---

### TC-03 — Список счетов

**Шаги:**
1. `GET /api/v1/cashflow/accounts`

**Ожидаемый результат:**
- HTTP 200
- Оба созданных счёта присутствуют в ответе
- У каждого счёта есть поле `current_balance`

---

### TC-04 — Попытка создать счёт с отрицательным балансом

**Шаги:**
1. `POST /api/v1/cashflow/accounts`
```json
{
  "name": "Тест",
  "account_type": "cash",
  "initial_balance": -100
}
```

**Ожидаемый результат:**
- HTTP 422 (Unprocessable Entity)
- Ошибка валидации Pydantic (initial_balance >= 0)

---

### TC-05 — Переименование счёта

**Шаги:**
1. `PATCH /api/v1/cashflow/accounts/{id}` с телом `{"name": "Касса офис"}`

**Ожидаемый результат:**
- HTTP 200
- `name` обновлён
- `current_balance` не изменился

---

### TC-06 — Деактивация счёта

**Шаги:**
1. `PATCH /api/v1/cashflow/accounts/{id}` с телом `{"is_active": false}`

**Ожидаемый результат:**
- HTTP 200
- `is_active` = false
- Счёт не отображается в `GET /cashflow/accounts` (без флага include_inactive)
- Счёт отображается при `GET /cashflow/accounts?include_inactive=true`

---

### TC-07 — Попытка создать счёт под ролью accountant (только admin)

**Предусловие:** авторизован как accountant  
**Шаги:**
1. `POST /api/v1/cashflow/accounts` с любыми данными

**Ожидаемый результат:**
- HTTP 403

## Блок 2: Категории (Categories)

### TC-08 — Получить список категорий

**Шаги:**
1. `GET /api/v1/cashflow/categories`

**Ожидаемый результат:**
- HTTP 200
- Присутствуют системные категории: "Оплата заказа", "Выплата зарплаты", "Начальный остаток", "Перевод между счетами"
- У системных категорий `is_system` = true

---

### TC-09 — Фильтрация категорий по типу

**Шаги:**
1. `GET /api/v1/cashflow/categories?transaction_type=expense`

**Ожидаемый результат:**

- Категория "Оплата заказа" (income) не присутствует

```json
}
```

**Ожидаемый результат:**
- HTTP 201
- `is_system` = false
- Категория появляется в списке

---

### TC-11 — Удаление пользовательской категории без операций

### TC-13 — Попытка удалить категорию с привязанными операциями

**Предусловие:** создана операция с пользовательской категорией

## Блок 3: Операции — Приход (Income)

### TC-14 — Создание ручного прихода (аренда от клиента)

**Предусловие:** создан счёт "Касса (наличные)", существует категория типа income  
**Шаги:**
1. `POST /api/v1/cashflow/transactions`
```json
{
  "transaction_type": "income",
  "account_id": 1,
  "category_id": <id категории income>,
  "amount": 5000,
  "description": "Предоплата от клиента"
}
```

**Ожидаемый результат:**
- HTTP 201
- `transaction_type` = "income"
- `amount` = 5000
- `account.name` соответствует счёту

---

### TC-15 — Проверка обновления баланса счёта после прихода

**Шаги:**
1. Записать `current_balance` счёта до операции (TC-01 = 35000)
2. Создать приход 5000 (TC-14)
3. `GET /api/v1/cashflow/accounts/{id}`

**Ожидаемый результат:**
- `current_balance` = 40000 (35000 + 5000)

---

### TC-16 — Создание прихода с датой в прошлом

**Шаги:**
1. `POST /api/v1/cashflow/transactions`
```json
{
  "transaction_type": "income",
  "account_id": 1,
  "category_id": <id>,
  "amount": 1000,
  "description": "Закрытие долга за февраль",
  "transaction_date": "2026-02-15T10:00:00"
}
```

**Ожидаемый результат:**
- HTTP 201
- `transaction_date` = "2026-02-15T10:00:00"

## Блок 5: Операции — Перевод (Transfer)

### TC-20 — Инкассация (перевод из кассы в банк)

**Предусловие:** есть два активных счёта  
**Шаги:**
1. Получить ID категории "Перевод между счетами"
2. `POST /api/v1/cashflow/transactions`
```json
{
  "transaction_type": "transfer",
  "account_id": 1,
  "to_account_id": 2,
  "category_id": <id transfer категории>,
  "amount": 20000,
  "description": "Инкассация"
}
```

**Ожидаемый результат:**
- HTTP 201
- `to_account` присутствует в ответе и соответствует счёту назначения
- Баланс счёта-источника уменьшился на 20000
- Баланс счёта-назначения увеличился на 20000

---

### TC-21 — Попытка перевода на тот же счёт

**Шаги:**
1. `POST /api/v1/cashflow/transactions`
```json
{
  "transaction_type": "transfer",
  "account_id": 1,
  "to_account_id": 1,
  "category_id": <id transfer>,
  "amount": 5000
}
```

**Ожидаемый результат:**
- HTTP 400
- Сообщение: "Счёт источника и счёт назначения не могут совпадать"

### TC-22 — Попытка перевода без to_account_id

## Блок 6: Автоматические операции

### TC-23 — Автоматический приход при оплате заказа

**Предусловие:**
- Создан хотя бы один активный счёт типа "cash"
- Существует заказ-наряд в статусе "ready_for_payment"

**Шаги:**
1. `POST /api/v1/orders/{order_id}/payments`
```json
{
  "order_id": <id>,
  "amount": 15000,
  "payment_method": "cash"
}
```
2. `GET /api/v1/cashflow/transactions?limit=5`

**Ожидаемый результат:**
- Платёж по заказу создан (HTTP 201)
- В списке операций кассы появилась автоматическая операция:
  - `transaction_type` = "income"
  - `amount` = 15000
  - `category.name` = "Оплата заказа"
  - `order_id` = id заказа
- Баланс кассового счёта увеличился на 15000

---

### TC-24 — Автоматический расход при выплате зарплаты

**Предусловие:**
- Создан хотя бы один активный счёт
- Существует рассчитанная зарплата (SalaryStatus.CALCULATED)

**Шаги:**
1. `POST /api/v1/salary/{salary_id}/pay`
2. `GET /api/v1/cashflow/transactions?limit=5`

**Ожидаемый результат:**
- Статус зарплаты стал PAID
- В операциях кассы появилась запись:
  - `transaction_type` = "expense"
  - `category.name` = "Выплата зарплаты"
  - `salary_id` = id расчёта
  - `amount` = сумма зарплаты

---

### TC-25 — Выплата зарплаты на конкретный счёт

**Шаги:**
1. `POST /api/v1/salary/{salary_id}/pay?account_id={bank_account_id}`
2. `GET /api/v1/cashflow/transactions?limit=5`

**Ожидаемый результат:**
- Расходная операция создана с `account_id` = указанный банковский счёт

---

### TC-26 — Нет кассового счёта — автооперация не создаётся (graceful)

**Предусловие:** все счета деактивированы или не созданы  
**Шаги:**
1. `POST /api/v1/orders/{order_id}/payments` — оплатить заказ

**Ожидаемый результат:**
- Оплата заказа прошла успешно (HTTP 201)
- Операция в кассе НЕ создана (счётов нет — не падает с ошибкой)

## Блок 7: Сводка (Summary)

### TC-27 — Сводка без фильтра по дате

**Шаги:**
1. `GET /api/v1/cashflow/summary`

**Ожидаемый результат:**
- HTTP 200
- `total_balance` = сумма `current_balance` всех активных счетов
- `total_income` = сумма всех приходов
- `total_expense` = сумма всех расходов
- `net_flow` = `total_income` - `total_expense`
- `accounts` — список с остатками по каждому счёту

---

### TC-28 — Сводка за текущий месяц

**Шаги:**
1. `GET /api/v1/cashflow/summary?date_from=2026-03-01T00:00:00&date_to=2026-03-31T23:59:59`

**Ожидаемый результат:**
- `total_income` учитывает только приходы за март 2026
- `total_expense` учитывает только расходы за март 2026
- `total_balance` не фильтруется по дате (всегда актуальный остаток)

---

### TC-29 — Арифметика баланса (интеграционный кейс)

**Шаги:**
1. Создать счёт с `initial_balance` = 10000
2. Добавить приход 5000
3. Добавить расход 3000
4. `GET /api/v1/cashflow/accounts/{id}`

**Ожидаемый результат:**
- `current_balance` = 10000 + 5000 − 3000 = **12000**
